# 项目解析与使用说明

## 1. 项目定位

这个项目最初是 Vue3 + Flask 博客，现在已经在向 ContextForge 方向升级：一个 AI 知识工作台。

它面向三类人：

- 访客：浏览公开知识文档。
- 普通用户：写文档、收藏、评论、维护自己的上下文包，并在 AI 对话中使用这些上下文。
- 管理员：维护用户、角色、AI 配置、Embedding 配置、系统自检和全站数据看板。

所以它不应该继续被理解为“博客加 AI 按钮”，更准确的产品形态是：

```text
知识库 + 上下文包 + AI 工作台 + 后台管理 + RAG 检索增强
```

## 2. 整体架构

```text
front/
  Vue 3 + Vite
  页面、路由、Pinia 状态、API SDK

backend/
  Flask
  蓝图路由、JWT 权限、SQLAlchemy 数据访问、AI 调用、RAG 索引

MySQL
  用户、角色、文档、评论、收藏、AI 配置、上下文包、RAG 分块
```

核心请求链路：

```text
Vue 页面
  -> front/src/api/modules/*
  -> front/src/api/index.ts Axios 封装
  -> Flask /api/*
  -> SQLAlchemy
  -> MySQL
```

AI 请求链路：

```text
AI 对话页面或 AI 起草按钮
  -> /api/ai/chat 或 /api/ai/generate-article
  -> 读取 active ai_configs
  -> 如果选择上下文包，先检索 RAG 片段
  -> 把少量命中片段和用户问题一起发送给模型
```

## 3. 后端模块

后端入口是 `backend/app.py`，主要蓝图如下：

| 文件 | 作用 |
| --- | --- |
| `routes/auth.py` | 注册、登录、用户信息、用户管理 |
| `routes/article.py` | 文档增删改查、点赞收藏、浏览量、评论 |
| `routes/context_pack.py` | 上下文包、资料来源、RAG 分块、Embedding、Markdown 导出 |
| `routes/ai.py` | AI 对话、流式对话、AI 起草、模型配置、Embedding 配置 |
| `routes/role.py` | 角色和权限管理 |
| `routes/system.py` | 健康检查、手动系统自检、自检历史 |
| `routes/upload.py` | 登录后图片上传 |

后端配置在 `backend/config.py`：

- 自动读取 `backend/.env`
- 支持 `DATABASE_URL` 或 Railway/云平台的 `MYSQL*` 环境变量
- `SECRET_KEY` 用于 JWT 签名

## 4. 数据库结构

`backend/init_db.py` 会初始化基础表：

| 表 | 作用 |
| --- | --- |
| `users` | 用户 |
| `roles` | 角色 |
| `permissions` | 权限点 |
| `role_permissions` | 角色权限关系 |
| `articles` | 知识文档，兼容旧文章模型 |
| `comments` | 文档讨论 |
| `article_likes` | 收藏/喜欢关系 |
| `ai_configs` | 聊天模型配置 |
| `context_packs` | 上下文包 |
| `context_pack_sources` | 上下文包资料来源 |

`context_pack.py` 还会按需维护：

- `context_pack_source_chunks`：RAG 检索分块
- `ai_embedding_configs`：Embedding 模型配置

## 5. 权限和数据可见性

当前规则：

- 未登录用户只能读取公开且已发布的文档。
- 私有文档只允许作者本人或文章管理员读取。
- 写文档、我的作品、我的喜欢、上下文包、AI 中心、仪表盘、个人中心都需要登录。
- 上传图片需要登录。
- AI 聊天、流式聊天、AI 起草和 AI 上下文记忆接口需要 `ai:access` 权限。
- AI 配置和 Embedding 配置需要 `ai:manage` 权限。
- 后台看板、详细系统自检和自检历史需要 `system:observe` 权限。
- `/api/health` 只保留最小健康状态，不再向访客暴露数据库、模型或 Key 配置细节。
- 管理员默认拥有全部权限。

默认管理员：

```text
admin / admin123
```

正式环境上线前应修改默认密码，并设置新的 `SECRET_KEY`。

## 6. 上下文包是什么

上下文包不是普通文件夹。它的作用是把一组资料组织成 AI 可复用的上下文资产。

典型用法：

- 毕设答辩资料包
- Vue3 学习资料包
- 项目部署排错包
- 简历项目介绍包
- 论文阅读包

上下文包里可以加入知识库文章作为资料来源。之后系统会把这些资料来源切成小块，用于检索。

## 7. RAG 检索原理

RAG 是 Retrieval-Augmented Generation，意思是“先检索，再生成”。

本项目的流程：

1. 用户把真实文档加入上下文包。
2. 点击重建 RAG 索引。
3. 后端把资料来源切成多个片段，写入 `context_pack_source_chunks`。
4. 用户提问时，系统先从片段里找最相关的几段。
5. 只把命中的少量片段发给 AI。
6. AI 基于这些片段回答。

这样做的好处：

- 不需要把整个上下文包塞进提示词。
- token 消耗更低。
- AI 更容易聚焦在相关材料上。
- 可以在页面上预览“本次会命中哪些片段”。

## 8. 关键词检索和语义检索

当前有两种检索：

| 类型 | 是否需要 Embedding | 是否额外耗费 token | 适合场景 |
| --- | --- | --- | --- |
| 关键词检索 | 不需要 | 不需要 embedding token | 标题、术语、明确关键词 |
| 语义检索 | 需要 | 生成向量时会耗费 embedding token | 问法和原文不完全一致 |

注意：

- 勾选语义检索不代表一定已经用上语义检索。
- 只有配置了 Embedding，并且给当前上下文包生成过当前模型的向量，语义检索才会启用。
- 否则系统会明确提示回退到关键词检索。

## 9. Markdown 导出是什么

上下文包导出 Markdown 不是“只获取摘要”。

它的意义是把当前上下文包整理成一个可复制、可归档、可发给其他 AI 或提交材料的文本包。

通常包含：

- 上下文包名称、用途、标签
- 包摘要和关键点
- 资料来源列表
- 每个资料来源的主要内容

它适合：

- 做项目资料归档
- 复制给外部 AI
- 写答辩材料
- 生成项目复盘
- 作为需求或论文阅读上下文

但在系统内 AI 对话时，不应该每次直接塞完整 Markdown。正确方式是 RAG：只检索相关片段。

## 10. 复制提示词和导出 Markdown 的区别

| 功能 | 目标 | 内容形态 | 适合用途 |
| --- | --- | --- | --- |
| 复制提示词 | 指挥 AI 完成任务 | 指令为主，资料为辅 | 让 AI 按某种方式分析、生成、提问 |
| 导出 Markdown | 归档或迁移上下文包 | 资料本身为主 | 存档、交接、发给其他工具 |

后续可以继续完善 Prompt Factory，让“复制提示词”生成更结构化的任务指令。

## 11. 如何使用核心功能

### 写一篇真实文档

1. 登录。
2. 进入 `/essays/write`。
3. 填写标题、类型、状态、可见性、来源链接、标签。
4. 手写正文，或点击 AI 起草。
5. 保存。

### 用 AI 起草

1. 进入新建文档页。
2. 点击 AI 起草。
3. 输入需求，例如“帮我生成一份 Flask JWT 权限设计文档”。
4. 生成后正文会写入编辑器，而不是原始 JSON。

### 建立上下文包

1. 进入 `/context-packs`。
2. 新建上下文包。
3. 从知识库选择文章加入。
4. 点击重建 RAG 索引。
5. 在 RAG 检索预览中输入问题，查看命中片段。

### 使用上下文包对话

1. 进入 `/ai-center/chat`。
2. 选择一个上下文包。
3. 输入问题。
4. 系统会先检索相关片段，再把片段和问题一起交给 AI。

### 启用语义检索

1. 进入 AI 中心。
2. 配置 Embedding provider、model、base_url、api_key。
3. 回到上下文包页面。
4. 对上下文包执行生成向量。
5. 再勾选语义检索。

## 12. 当前系统还没完全完成的地方

已经完成或正在形成闭环：

- 登录、用户、角色、权限基础能力
- 文档 CRUD、评论、收藏、浏览量
- AI 起草、AI 聊天、上下文包接入聊天
- 上下文包 CRUD、资料来源、Markdown 导出
- RAG 分块索引、关键词检索、Embedding 配置和语义检索开关
- 手动系统自检，已移除自动自我进化调度
- 部分后台页面真实数据化

仍建议继续完善：

- 后台入口按管理员和普通用户拆分得更清楚。
- 用户管理、角色管理页面需要进一步检查按钮和权限反馈。
- Prompt Factory 还未形成完整页面。
- 文档详情页右侧 AI 洞察可以继续增强。
- 大包体积需要后续做按路由拆包和依赖优化。
- 生产环境需要 HTTPS、强密钥、默认密码修改和数据库备份策略。
